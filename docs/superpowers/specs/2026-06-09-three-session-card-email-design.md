# 三时段卡片邮件 + 自主决策指令 — 设计文档

**日期**: 2026-06-09
**状态**: 已与用户确认，待落地
**关联**: `SKILL.md`（运行时提示词）、`scripts/send_email.py`（渲染器，已具备能力）

## 1. 背景与目标

用户（收件人「徐勇」）自认对投资"啥也不懂"，要求本 skill 走**自主经营**模式：

1. 引擎自主决策，**不再反复询问用户**该不该买/卖；
2. 分析+决策完成后，直接给一条**可照抄的下单指令**（买什么、多少钱、支付宝搜什么代码）；
3. 每日按**开盘 / 盘中 / 盘尾**三个时段，各发一封**卡片式 HTML 邮件**（样式见用户提供的截图）；
4. 目标：在引擎风控规则内追求收益最大化。

### 执行定调（2026-06-09 用户最终确认）

- 用户明确："你说买什么 = 已下单，我必跟单，**无需任何支付宝交互确认**"。因此 **AI 的盘尾决策视同成交**：决策→自动记账→卡片邮件通知，**当作已成交，不加"你还需手动点"之类免责话**。
- "自主"= 决策自主 + 下单即记账（写 DB）。用户是恒定可靠的执行臂，AI 不再纠结"无支付宝 API"。
- 唯一不能做的是**对用户谎称保证盈利**——目标是引擎规则内收益最大化，风控闸门只为保本、服务长期复利。

## 2. 关键发现

`scripts/send_email.py` 的 `markdown_to_html()` **已经支持**截图那套卡片 DSL，无需改渲染器：

| DSL 块 | 渲染结果 |
|---|---|
| `:::card` | 顶部大盈亏数字 + 一行 stats（市值/昨日/累计/%）|
| `:::action` | 黄色操作提示框（左侧橙色竖条）|
| 持仓表格 `\| 基金 \| 今日 \| 今日盈亏 \| 昨日盈亏 \| 累计 \| 持有 \|` | 持仓卡片行 |
| `:::blocks` | 大盘/板块热力块（按涨跌幅着色、自动分层）|
| `:::timeline` | 近期操作时间线（买绿卖红圆点）|

**Gap**：当前 `SKILL.md` 完全没提这套 DSL，§5.1 午报模板用的是普通 markdown 标题/表格，发出来是朴素样式。且现有只有「午报 14:30 + 晚报 21:00」，**无三时段概念**。

**已知小瑕疵（保持现状不改）**：`:::card` 顶部大数字对正数用绿色（与"红涨绿跌"惯例相反），但截图正是此渲染器生成、用户认可，故不动。

## 3. 三时段设计

三时段**共用同一套卡片结构**，仅顶部数字语义与黄框文案随时段变化。对齐场外基金"按当日收盘净值成交、约 15:00 截单"的规则：

| 模块 | 开盘 ~9:30 | 盘中 ~13:00 | 盘尾 ~14:48 |
|---|---|---|---|
| 顶部 `:::card` | 昨日收盘盈亏 + "今日计划" | 今日估算盈亏（实时） | 今日估算盈亏（尾盘，截图样式）|
| 黄框 `:::action` | **今日操作计划**（预告今天打算买/卖什么）| 盘中信号（是否触发临时低吸/止损）| **最终下单指令**（最重要：今天买X¥Y / 全部持有）|
| 持仓表格 | 昨收基准 | 实时估值 | 尾盘估值 |
| `:::blocks` | 昨收 + 外盘提示 | 实时板块 | 实时板块 |
| `:::timeline` | ✅ | ✅ | ✅ |

**盘尾 14:48 是真正的下单决策窗口**，三封里最重要。开盘/盘中是预告与盯盘。

## 4. 数据接线（运行时 Claude 按此组装卡片 markdown）

| 卡片槽位 | 数据来源命令 | 字段 |
|---|---|---|
| `:::card` 顶部数字/stats | `fetch_fund.py portfolio-check --account 主线` | `total_today_pnl`、持仓市值、累计盈亏、累计% |
| 持仓表格（今日%/今日盈亏/累计%）| `fetch_fund.py portfolio-check` | 每只 `today_pnl`、`gszzl`、累计% |
| `:::blocks` 大盘热力 | `fetch_fund.py sectors` + `fetch_fund.py indices` | 板块名+涨跌%、指数名+涨跌% |
| `:::timeline` 近期操作 | `db.py trades --account 主线 --limit 8` | 日期、买/卖、名称、金额 |
| `:::action` 操作指令 | `decide.py run --account 主线 --format json` | `actions[].reason_zh`、`suggested_amount`、`code` |

**决策权威来源仍是 `decide.py` 引擎**——自主≠失控：现金储备、禁追涨、止损、仓位上限等硬规则照常拦截（见 `blocked_actions[]`）。黄框指令必须忠实反映引擎 `actions[]`，不得 Claude 自行加减。

## 5. 自主指令口吻

黄框 `:::action` 从"建议"改为"指令"，并补足可执行细节：

- 有买入：`今日操作：支付宝搜 512480，买入「半导体ETF国联安」¥2000。其余持有。`
- 有卖出：`今日操作：卖出「广发纳斯达克100C」约 600 份（¥5000）止盈。`
- 无操作：`今日不操作，全部持有。{引擎给的市场叙事一句}。`

指令金额/份额取引擎 `suggested_amount` / `suggested_shares`。份额按用户实际成交净值换算的提示可附在指令里。

### 5.1 执行/记账模型（2026-06-09 用户确认）

用户会**完全跟着 AI 买**，授权 AI **自动记账**：

- **只有盘尾 14:48 这一封负责执行 + 记账**。开盘/盘中只发信息，绝不写 DB。
- 盘尾引擎给出未被拦截的 buy/sell 时，**不必等用户回"买了"**，直接走 §四：`db.py add/remove-position` → `add-order` → `trade-notify`，份额按当日估算净值 gsz 换算，cost 填 gsz。
- **净值校准**：晚间实际净值公布后，把该笔 cost_nav 校准为实际净值（误差 0.5-2%）。
- **容错回退**：用户没跟单/买了别的，会回来用 skill"调整持仓"，据实校正；账本最终以用户实告为准。
- **物理边界不变**：仍无法替用户在支付宝点"买入"；自动记账基于"用户完全跟单"的授权前提。
